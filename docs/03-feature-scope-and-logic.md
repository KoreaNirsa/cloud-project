# 필수 기능 및 필수 로직 정의서

이 문서는 **무조건 구현해야 하는 기능**과  
**해당 기능이 제대로 동작하기 위해 필요한 핵심 로직**을 정리한 문서입니다.

---

## 1. 범위 우선순위

| 우선순위 | 의미 |
|---|---|
| P0 | 반드시 구현 |
| P1 | 시간 되면 구현 |
| P2 | 발표용 확장 포인트 |

---

## 2. P0 필수 기능 목록

## 2.1 인증 / 권한
- 로그인
- 현재 로그인 사용자 조회
- 역할별 접근 제어 (`EMPLOYEE`, `MANAGER`, `ADMIN`)

### 필수 로직
- 로그인 ID / 비밀번호 검증
- 인증 성공 시 토큰 또는 세션 발급
- 역할에 맞지 않는 API는 403 반환

### 화면 연결
- 로그인 페이지
- 상단 헤더 사용자 정보
- 역할별 메뉴 노출

---

## 2.2 직원 관리
- 직원 목록 조회
- 직원 상세 조회
- 직원 등록
- 직원 수정
- 직원 상태 변경(ACTIVE / INACTIVE)

### 필수 로직
- 사번 중복 방지
- 로그인 ID 중복 방지
- 이메일 중복 방지
- 부서 필수 지정
- 매니저는 같은 회사 내 직원만 지정 가능

### 화면 연결
- 직원 목록 페이지
- 직원 상세 페이지
- 직원 등록/수정 모달 또는 폼

---

## 2.3 부서 관리
- 부서 목록 조회
- 부서 등록
- 부서 수정

### 필수 로직
- 부서 코드 중복 방지
- 부서명 중복 방지(권장)
- 직원이 소속된 부서를 삭제하는 기능은 MVP에서 제외하거나 비활성화 권장

### 화면 연결
- 부서 관리 페이지

---

## 2.4 출근 / 퇴근
- 출근 처리
- 퇴근 처리
- 내 월간 근태 조회
- 팀 근태 조회(매니저/관리자)

### 필수 로직
- 하루 1개 근태 레코드만 생성
- 출근하지 않은 상태에서는 퇴근 불가
- 이미 퇴근한 경우 중복 퇴근 불가
- 출근 시각 기준으로 `PRESENT` / `LATE` 계산
- 휴가 승인된 날짜에는 출근 불가 처리 권장

### 추천 지각 기준
- 09:15 이전 또는 같음 → `PRESENT`
- 09:15 이후 → `LATE`

### 화면 연결
- 대시보드 출근/퇴근 카드
- 내 근태 월간 목록
- 팀 근태 표

---

## 2.5 휴가 신청 / 승인
- 휴가 종류 조회
- 휴가 신청
- 내 휴가 목록 조회
- 휴가 취소
- 승인 대기 목록 조회
- 승인 / 반려 처리

### 필수 로직
- 시작일 <= 종료일 검증
- 중복 휴가 기간 방지
- 연차 차감 휴가인 경우 잔여 연차 검증
- 승인자는 팀원 또는 권한 있는 관리자여야 함
- 이미 승인/반려/취소된 건은 재승인 불가
- 본인이 올린 신청 건은 본인이 승인하지 못하도록 제한 권장
- 취소는 `REQUESTED` 또는 시작일 이전의 `APPROVED`만 허용 권장

### 휴가 일수 계산 권장
- `영업일 수 = 기간 내 날짜 - 주말 - 등록된 공휴일`
- 구현 부담이 크면 1차는 단순 일수 계산 후, 2차 개선

### 화면 연결
- 휴가 신청 폼
- 내 휴가 목록
- 승인 대기 목록
- 팀 대시보드

---

## 2.6 대시보드
- 내 오늘 근태 상태
- 이번 달 출근일 수
- 남은 연차
- 승인 대기 건수(매니저/관리자)
- 오늘 팀 휴가자 수(매니저/관리자)

### 필수 로직
- 여러 테이블(employee, attendance, leave_request)를 조합한 조회
- 사용자 역할에 따라 응답 필드 차등 제공 가능
- 대시보드는 조회 성격이 강하므로 JPQL/Querydsl/Native SQL 중 가장 쉬운 방식 선택

---

## 2.7 공휴일 관리
- 공휴일 목록 조회
- 공휴일 등록
- 공휴일 수정/삭제(선택)

### 필수 로직
- 같은 날짜 중복 등록 방지
- 휴가 일수 계산 시 참조 가능해야 함

---

## 2.8 클라우드 배포
- 프론트엔드 S3 + CloudFront 배포
- 백엔드 EC2 배포
- RDS 연결

### 필수 로직
- 환경 변수 분리
- CORS 설정
- 실제 배포 주소에서 API 연결 확인
- 최소 1개의 데모 계정으로 발표 가능 상태 유지

## 3. 핵심 로직 상세

## 3.1 로그인 로직
```text
로그인 요청
→ loginId 조회
→ passwordHash 검증
→ 성공 시 access token 발급 또는 세션 생성
→ 프론트는 사용자 role 기반으로 메뉴 렌더링
```

### 실패 케이스
- 존재하지 않는 loginId
- 비밀번호 불일치
- 비활성화된 직원(INACTIVE)

---

## 3.2 출근 로직
```text
오늘 날짜 기준 attendance 조회
→ 이미 있으면 중복 출근 실패
→ 승인된 휴가 여부 확인
→ 현재 시각으로 attendance 생성
→ 출근 시각 기준 status 계산
```

### 실패 케이스
- 이미 출근한 경우
- 승인된 휴가 날짜인 경우
- 비활성 사용자

---

## 3.3 퇴근 로직
```text
오늘 attendance 조회
→ 없으면 실패
→ checkOutAt 이미 있으면 실패
→ 현재 시각 저장
→ 근무 시간 계산(선택)
```

### 실패 케이스
- 출근 기록 없음
- 이미 퇴근 처리됨

---

## 3.4 휴가 신청 로직
```text
휴가 신청 요청
→ 날짜 범위 검증
→ 중복 기간 검증
→ 휴가 종류 조회
→ 연차 차감 여부 확인
→ 잔여 연차 계산
→ 부족하면 실패
→ approver 지정
→ 상태 REQUESTED 로 저장
```

### 실패 케이스
- 시작일 > 종료일
- 본인 휴가와 기간 겹침
- 연차 부족
- approver 없음
- 비활성 사용자

---

## 3.5 휴가 승인 로직
```text
승인 요청
→ leaveRequest 조회
→ 상태가 REQUESTED 인지 확인
→ approver 권한 확인
→ 승인 가능 범위(본인 팀원인지) 확인
→ status = APPROVED
→ decisionAt 저장
```

### 실패 케이스
- 이미 승인/반려/취소된 건
- 권한 없는 사용자
- 자기 자신의 신청 건 승인 시도
- 팀 외 직원 승인 시도

---

## 3.6 휴가 반려 로직
```text
반려 요청
→ leaveRequest 조회
→ 상태가 REQUESTED 인지 확인
→ approver 권한 확인
→ status = REJECTED
→ rejectionReason 저장
→ decisionAt 저장
```

---

## 3.7 남은 연차 계산 로직
권장 계산식:

```text
remainingAnnualLeave
= annualLeaveQuota
- sum(approved leave days where deductsFromAnnual = true)
```

### 주의
- `REQUESTED` 상태는 차감하지 않음
- `APPROVED`만 차감
- `CANCELLED`는 차감하지 않음
- 반차를 넣는다면 0.5 단위 허용 필요

## 4. 확장 포인트(P1 / P2)

## 4.1 P1 권장 확장
- 직원 검색/필터(부서, 상태)
- 휴가 신청 수정(REQUESTED 상태만)
- 공휴일 수정/삭제
- 프로필 사진 업로드
- CSV 내보내기
- 관리자 전체 근태 조회

## 4.2 P2 발표용 확장
- 반차 / 시간차
- 이메일 알림
- Slack Webhook 알림
- 조직도 화면
- 근태 통계 차트
- 감사 로그(audit log)
- 급여 모듈 설계 초안

## 5. 반드시 지켜야 할 비즈니스 규칙 요약

| 구분 | 규칙 |
|---|---|
| 직원 | 사번, loginId, 이메일은 중복 불가 |
| 직원 | 비활성 직원은 로그인 불가 |
| 근태 | 하루에 한 번만 출근 가능 |
| 근태 | 퇴근은 출근 후에만 가능 |
| 근태 | 출근 시각 기준 상태 계산 |
| 휴가 | 날짜 범위가 올바라야 함 |
| 휴가 | 중복 휴가 신청 금지 |
| 휴가 | 연차 부족 시 신청 금지 |
| 휴가 | 승인자는 권한과 범위를 만족해야 함 |
| 휴가 | 상태 전이는 REQUESTED 기준으로만 승인/반려 가능 |
| 공휴일 | 같은 날짜 중복 등록 금지 |

## 6. 학생팀을 위한 구현 팁

### 팁 1. 급여는 빼라
급여를 넣는 순간 규칙이 급격히 복잡해집니다.  
이번 프로젝트에서는 HR Core만 구현하는 것이 완성도 면에서 훨씬 유리합니다.

### 팁 2. 연차 계산은 “조회 시 계산”으로 시작하라
테이블 캐시를 만들기보다 먼저 서비스 레이어에서 계산하세요.  
데이터 불일치를 줄일 수 있습니다.

### 팁 3. 출근/퇴근은 당일만 허용하라
임의 날짜 입력까지 열어두면 예외 케이스가 급격히 늘어납니다.

### 팁 4. 승인 흐름은 Leave에만 집중하라
전자결재 전체를 구현하려고 하지 말고,
이번 프로젝트에서는 **휴가 승인** 한 가지 흐름만 끝까지 만드세요.
