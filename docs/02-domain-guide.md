## 1. HR 시스템이란?

HR 시스템은 회사의 사람(직원)과 관련된 업무를 관리하는 시스템입니다.

대표적인 하위 영역은 아래와 같습니다.

| 영역 | 설명 |
|---|---|
| 인사 기본 정보 | 직원, 부서, 직급, 입사일, 재직 상태 |
| 근태 | 출근, 퇴근, 지각, 결근, 재택 등 |
| 휴가 | 연차, 병가, 반차, 휴가 승인 |
| 급여 | 월급, 수당, 공제 |
| 평가 | 성과 리뷰, 피드백 |
| 채용 | 지원자 관리, 인터뷰 일정 |

이번 프로젝트는 이 중에서 **인사 기본 정보 + 근태 + 휴가 승인**에 집중합니다.

---

## 2. 이번 프로젝트의 도메인 범위

```text
직원(Employee)
  ├─ 부서(Department)에 소속됨
  ├─ 매니저(Manager)를 가질 수 있음
  ├─ 근태(Attendance)를 기록함
  └─ 휴가(Leave Request)를 신청함
```

---

## 3. 주요 사용자 역할

## 3.1 Employee (일반 직원)
- 로그인
- 내 정보 조회
- 출근/퇴근
- 내 근태 조회
- 휴가 신청
- 내 휴가 목록 조회
- 내 휴가 취소

## 3.2 Manager (팀장 또는 조직 책임자)
- 팀원 조회
- 팀원 근태 확인
- 팀원 휴가 승인/반려
- 팀 대시보드 조회

## 3.3 Admin / HR (인사 담당자)
- 직원 등록/수정
- 부서 관리
- 공휴일 관리
- 전체 데이터 조회
- 필요한 경우 승인 대체 처리

---

## 4. 핵심 개념 설명

## 4.1 Employee (직원)
시스템의 가장 기본이 되는 사용자입니다.

주요 정보
- 사번(employeeNo)
- 로그인 ID(loginId)
- 이름(name)
- 이메일(email)
- 부서(department)
- 매니저(manager)
- 역할(role)
- 재직 상태(employmentStatus)
- 연차 부여 일수(annualLeaveQuota)

> 이번 MVP에서는 “직원 = 시스템 사용자”로 간주합니다.  
> 즉, 직원이 로그인합니다.

---

## 4.2 Department (부서)
회사의 조직 단위입니다.

예시
- HR
- Engineering
- Design
- Sales

이번 프로젝트에서는 **직원은 한 개 부서에 소속**된다고 가정합니다.

---

## 4.3 Attendance (근태)
직원이 언제 출근했고 언제 퇴근했는지 기록하는 데이터입니다.

이번 MVP에서는 아래 정도만 다룹니다.

- 오늘 출근
- 오늘 퇴근
- 월간 내역 조회
- 상태 표시

근태 상태 예시
- `PRESENT`: 정상 출근
- `LATE`: 지각
- `REMOTE`: 재택(확장)
- `ON_LEAVE`: 휴가 중(주로 계산 결과로 표현)
- `ABSENT`: 무단 결근(주로 계산 결과로 표현)

> `ON_LEAVE`, `ABSENT`는 실제로 attendance 테이블에 직접 저장하지 않고  
> 화면이나 조회 API에서 계산해서 보여주는 방식도 충분히 가능합니다.

---

## 4.4 Leave Request (휴가 신청)
직원이 휴가를 쓰기 위해 올리는 신청 데이터입니다.

주요 정보
- 신청자
- 휴가 종류
- 시작일 / 종료일
- 휴가 일수
- 사유
- 승인자
- 상태

휴가 상태
- `REQUESTED`: 신청 완료, 아직 승인 전
- `APPROVED`: 승인 완료
- `REJECTED`: 반려
- `CANCELLED`: 신청 취소

---

## 4.5 Leave Type (휴가 종류)
휴가의 종류입니다.

예시
- `ANNUAL`: 연차
- `SICK`: 병가
- `OFFICIAL`: 공가
- `HALF_DAY`: 반차(확장)

모든 휴가가 연차 차감 대상은 아닙니다.  
예를 들어 병가/공가는 회사 정책에 따라 연차 차감이 아닐 수 있습니다.

그래서 `leave_type`에는 다음과 같은 정보가 필요합니다.

- code
- name
- deductsFromAnnual (연차 차감 여부)

---

## 4.6 Holiday (공휴일)
회사가 쉬는 날입니다.

휴가 계산이나 달력 화면에서 참조할 수 있습니다.

예시
- 신정
- 설날
- 어린이날
- 회사 창립기념일

---

## 5. 이번 프로젝트에서 중요한 업무 규칙

## 5.1 직원-부서 규칙
- 직원은 반드시 1개 부서에 속한다.
- 매니저도 직원이다.
- 직원은 선택적으로 매니저를 가질 수 있다.
- 같은 회사 안에서 사번은 중복되면 안 된다.

## 5.2 출근/퇴근 규칙
- 하루에 출근은 1번만 가능하다.
- 퇴근은 출근 이후에만 가능하다.
- 이미 퇴근한 날에는 다시 퇴근할 수 없다.
- 당일 기준으로만 출근/퇴근하도록 제한하는 것이 안전하다.

## 5.3 휴가 신청 규칙
- 시작일이 종료일보다 늦을 수 없다.
- 이미 승인되었거나 신청 중인 휴가 기간과 겹치면 안 된다.
- 연차 차감 휴가는 잔여 연차가 부족하면 신청할 수 없다.
- 승인자는 본인 신청을 승인할 수 없게 막는 것을 권장한다.
- 매니저는 자기 팀원만 승인 가능하게 구현하는 것이 좋다.

## 5.4 공휴일 규칙
- 휴가 일수 계산 시 공휴일 제외 여부는 팀 합의가 필요하다.
- 이번 문서에서는 **주말 + 등록된 공휴일을 제외한 영업일 기준 계산**을 권장한다.
- 시간이 부족하면 MVP에서는 “단순 날짜 차이 계산”으로 먼저 구현하고 이후 개선해도 된다.

---

## 6. 상태 전이(State Transition)

## 6.1 휴가 신청 상태 전이

```text
REQUESTED
  ├─ approve → APPROVED
  ├─ reject  → REJECTED
  └─ cancel  → CANCELLED
```

### 권장 제약
- `APPROVED` 후에는 일반 수정 불가
- `REJECTED` 후에는 재신청하도록 유도
- `CANCELLED`는 시작일 이전까지만 허용하는 방식 추천

---

## 6.2 근태 상태 계산 예시

```text
출근 시각 <= 09:15 → PRESENT
출근 시각 >  09:15 → LATE
```

> 시간 규칙은 팀이 합의해서 자유롭게 바꿔도 됩니다.  
> 중요한 것은 **규칙이 문서화되어 있고 일관되게 동작하는 것**입니다.

## 7. 학생들이 자주 헷갈리는 포인트

### Q1. 직원과 유저를 따로 나눠야 하나요?
- 실무에서는 `User`(인증)와 `Employee`(인사 데이터)를 분리하는 경우가 많습니다.
- 하지만 이번 프로젝트에서는 일정이 짧기 때문에 **하나의 employee 테이블에 로그인 필드까지 넣는 단순화 모델**도 괜찮습니다.
- 문서의 DDL은 **employee 안에 loginId/passwordHash를 포함하는 단순화 모델** 기준입니다.

### Q2. 연차 잔여 개수는 컬럼으로 저장하나요?
- 초보 프로젝트에서는 `annualLeaveQuota`를 employee에 두고,
- 남은 연차는 **승인된 휴가 합계를 계산**해서 보여주는 방식이 더 안전합니다.
- 저장형 캐시는 나중에 확장 가능합니다.

### Q3. 결근은 테이블에 저장해야 하나요?
- 꼭 그럴 필요는 없습니다.
- 특정 날짜에 출근 기록도 없고 휴가도 아니면 화면에서 `ABSENT`로 계산해 보여줄 수 있습니다.

### Q4. 반차는 꼭 해야 하나요?
- 아닙니다.
- 반차/시간차는 계산 로직이 복잡해집니다.
- 이번 문서에서는 **확장 포인트**로 둡니다.

## 8. 이번 프로젝트의 좋은 데모 포인트

아래 흐름이 자연스럽게 나오면 프로젝트가 좋아 보입니다.

1. 직원 등록
2. 직원 로그인
3. 출근
4. 휴가 신청
5. 매니저 승인
6. 대시보드/목록 반영
7. 배포 URL에서 동작 확인

## 9. 프로젝트 범위를 지키기 위한 원칙

아래는 이번 프로젝트에서 **하지 않아도 되는 것**입니다.
추후 프로젝트를 확장하고 싶을 때 적용하시면 됩니다.

- 법정 근로시간 계산 완벽 대응
- 다양한 휴가 단위(시간차/분차)
- 급여/수당 계산
- 조직도 시각화
- 메일 서버 연동
- 다국어 지원
- 다중 회사(멀티 테넌시)
