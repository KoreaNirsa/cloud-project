# 팀 운영 계획 및 개발 일정

이 문서는 **학생 풀스택 3명**이  
짧은 기간 안에 충돌을 줄이고 끝까지 완성하기 위한 운영 계획입니다.

---

## 1. 기본 전략

### 핵심 원칙
- 한 명이 프론트만, 다른 한 명이 백만 맡는 구조보다  
  **모듈 단위(기능 단위)로 프론트+백엔드를 같이 맡는 방식**이 더 좋습니다.
- 공통 설계는 함께 하고, 구현은 모듈 소유권을 나눕니다.
- `ERD`, `API`, `공통 응답 규격`을 먼저 확정한 뒤 개발합니다.

## 4. 권장 개발 순서

### Step 1. 설계 고정
- 기획서 읽기
- 도메인 가이드 읽기
- ERD 확정
- API 명세 확정
- 역할 분담 확정

### Step 2. 공통 기반 만들기
- 백엔드 프로젝트 생성
- 프론트 프로젝트 생성
- 공통 응답 형식
- 공통 예외 처리
- 인증 골격
- 공통 레이아웃

### Step 3. 모듈 구현
- A: 직원/부서
- B: 근태
- C: 휴가/승인/공휴일

### Step 4. 연동
- 프론트 ↔ 백 연결
- 권한 처리
- 대시보드 연결

### Step 5. 배포 / QA
- AWS 배포
- 시나리오 테스트
- 발표 준비

## 10. 역할 분담 시 주의할 점

### 하지 말아야 할 방식
- 한 명이 “API 전부” 담당
- 한 명이 “UI 전부” 담당
- 한 명이 “배포 전부” 담당

이렇게 하면 병목이 생기고 일정이 밀립니다.

### 좋은 방식
- 모듈 단위로 화면/API/DB를 함께 본다
- 공통 영역은 짧게 합의하고 빠르게 구현한다
- 통합은 일찍 시작한다

- [ ] API 동작

- [ ] 예외 케이스 처리
